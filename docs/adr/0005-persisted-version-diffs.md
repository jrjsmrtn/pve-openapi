# 5. Persisted Version Diffs

## Date

2026-03-27

## Status

Accepted

## Context

The `PveOpenapi.Diff` module computes diffs between PVE API versions at runtime by comparing endpoint sets and parameter schemas. This works well for programmatic use within Elixir, but:

- Consumer projects may want to inspect diffs without running Elixir code
- Community users browsing the repository on Codeberg/GitHub should be able to see what changed between versions
- Pre-computed diffs avoid redundant work when the same diff is queried repeatedly
- JSON diffs can be consumed by tools in any language

## Decision

We will persist version diffs as JSON files in `specs/diffs/`, generated by `mix pve_openapi.diff --all` and included in `make setup`.

**File naming**: `diff-{from}-{to}.json` (e.g., `diff-8.3-8.4.json`)

**JSON format**:

```json
{
  "from": "8.3",
  "to": "8.4",
  "generated_at": "2026-03-27",
  "summary": {
    "added": 0,
    "removed": 0,
    "parameter_changes": 5,
    "breaking": 0
  },
  "added_endpoints": [],
  "removed_endpoints": [],
  "parameter_changes": [
    {
      "path": "/access/domains",
      "method": "post",
      "changes": [
        {"type": "constraint_changed", "name": "groups-claim", "field": "pattern", "from": "...", "to": "..."}
      ]
    }
  ],
  "breaking_changes": []
}
```

**Parameter change types**:
- `param_added` — new parameter (with `required` boolean)
- `param_removed` — parameter removed (breaking)
- `type_changed` — parameter type changed (breaking, with `from`/`to`)
- `became_required` — optional parameter became required (breaking)
- `constraint_changed` — constraint modified (with `field`, `from`, `to`)

**Breaking change classification**: `endpoint_removed`, `param_removed`, `type_changed`, and `became_required` are considered breaking.

**Diff files are `.gitignored`** — like specs, they are reproducible artifacts generated from the extraction pipeline. They are generated for consecutive version pairs only (not all permutations).

## Consequences

### Positive

- Machine-readable API evolution history accessible without Elixir
- Consistent with the `.gitignored specs` approach (ADR-4)
- `make setup` produces a complete picture (specs + metadata + diffs)
- Breaking change detection enables informed consumer migration

### Negative

- Additional generated artifacts to manage (mitigated: `make clean` handles them)
- Only consecutive pairs are pre-computed (arbitrary pairs still require runtime computation)
